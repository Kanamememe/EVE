EVE Chat Adapter Map v0.2
=========================

已確認的核心掛載點
------------------
1. API 請求
   - EVE Chat 使用 window.fetch 呼叫 generativelanguage.googleapis.com。
   - adapter.js 只攔截 Gemini generateContent / streamGenerateContent 請求。
   - 背景資料被附加到 systemInstruction，不改動原本聊天歷史。

2. AI 回覆入口
   - 頁面提供 window.triggerSmartReply()。
   - 聊天工具列也有 onclick="triggerSmartReply()" 的按鈕。
   - 主動訊息優先呼叫函式，找不到時才點擊按鈕。

3. 使用者輸入
   - 聊天輸入框：#api-chat-input。
   - adapter 監聽 Enter 與送出按鈕，通知 proactive 模組重設閒置時間。

4. 地點設定
   - #character-city-name
   - #character-city-prototype
   - 欄位變更後呼叫 EVEWeather.syncFromLocationForm()。

5. 模組載入順序
   - js/weather.js
   - js/proactive.js
   - js/adapter.js
   這三個 script 已放在 index.html 原本大型核心 script 之後、body 結束前。

事件流
------
使用者送出訊息
  -> adapter 偵測
  -> eve:user-message-sent
  -> EVEProactive.markUserInteraction()

一般 AI 回覆
  -> EVE Chat 建立 Gemini request
  -> adapter fetch hook
  -> 收集 weather / activity / 其他 provider
  -> 注入 systemInstruction
  -> eve:ai-request
  -> Gemini 回覆
  -> eve:ai-response 或 eve:ai-error

主動訊息
  -> proactive 排程完成
  -> eve:proactive-message-request
  -> adapter 設定一次性主動訊息指令
  -> triggerSmartReply()
  -> 下一個 Gemini request 注入主動訊息背景
  -> 請求結束後自動清除一次性指令

給後續模組的穩定介面
--------------------
EVEAdapter.registerContextProvider(id, function, { priority })
EVEAdapter.unregisterContextProvider(id)
EVEAdapter.getPromptContext()
EVEAdapter.requestProactiveMessage(payload)
EVEAdapter.getDiagnostics()

後續 memory.js、timeline.js、world.js 不必再修改 fetch 或 index.html。
